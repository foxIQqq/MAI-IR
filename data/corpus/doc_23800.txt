Релиз драйвера radeonhd 1.3.0 и план подготовки релиза X Server 1.8.0

После 5 месяцев разработки увидел свет релиз свободного X.Org драйвера для видеокарт ATI, построенных на базе чипсетов r5xx/r6xx/r7xx, - xf86-video-radeonhd 1.3.0. Драйвер создан независимыми разработчиками при поддержке компании AMD и Novell, а также благодаря открытию компанией AMD спецификаций на свои графические чипсеты. Драйвер совместим с RandR 1.3, поддерживает механизмы 2D и Xv (video) акселерации для всех поддерживаемых GPU. Функции 3D акселерации доступны только через Mesa для r5xx/rs690 GPU (X1xxx) и скоро будут доведены до рабочего состояния для r6xx/r7xx GPU (HD2xxx-HD4xxx). В новой версии отмечены следующие новшества: Добавлена поддержка карт RV740, M92, M93, M97; Добавлена поддержка HDMI аудио для карт на базе RS690 и R700. Реализованы функции управления питанием видеокарт; Реализован корректный анализ PowerPlay таблиц для AtomBIOS; По умолчанию активирована поддержка 2D-акселерации EXA, за исключением карт на базе RV740; Устранены проблемы с управлением фоновой подсветкой, реализация теперь совместима с xbacklight 1.1.1; Пересмотрены настройки системы управления памятью и методы управления курсором; Добавлена поддержка выборочных цветовых пространств при задействовании XVideo акселерации на картах R6xx/R7xx; Устранены все причины вывода предупреждений компилятора во время сборки. Кроме того, опубликован план подготовки X Server 1.8.0. Релиз намечен на 31 марта 2010 года, полная заморозка кода состоится 28 февраля, а переход на стадию прекращения добавления новшеств будет осуществлен 31 декабря 2009 г. В X Server 1.8.0 ожидается интеграция подсистемы XKB2, улучшение работы X Input 2 и оптимизация производительности архитектуры акселерации EXA. Напомню, что введена в силу новая схема разработки X.Org, основанная на жестком 6-месячном цикле подготовки релизов и ограничении доступа к master-ветке в репозитории исходных текстов, право выполнения коммитов в которой будет предоставлено только команде разработчиков, непосредственно отвечающих за выпуск релиза. Остальные разработчики должны производить изменения только в отдельных ветках репозитория, выделенных для каждой подсистемы X-сервера. Процесс разработки теперь разбит на три фиксированные стадии: добавление новых возможностей (3 месяца), выявление и исправление ошибок (2 месяца) и заморозка перед релизом (1 месяц).
